原因判断

结合注册、Key、消费、支付和错误日志，当前损耗更像新手路径和试用门槛问题。

  • 1. 注册后接入门槛高 近 7 天新增用户里 190 人没有创建 Key。用户可能看到了注册入口，但没有被顺畅带到“创建 Key、复制 Base URL、选择模型、完成一次请求”的路径。
  • 2. Key 到首次成功调用存在明显断点 全量 100 人有 Key 但无成功消费，近 7 天新增里有 94 人处在这个状态。API Key 的 accessed_time 均有记录，说明用户可能已经触达 Key，但没有完成有效请求。
  • 3. 新用户缺少直接试用额度 生产配置显示 QuotaForNewUser = 0，QuotaForInviter = 1,000,000。直注册用户没有新手额度，会阻断首次调用；当前 131 人处于无余额且无 active 订阅状态。
  • 4. 支付放弃率偏高 全量订单 107 笔中 expired/failed 54 笔。用户有支付意图但没有完成支付，可能与支付页说明、支付有效期、回跳确认或价格认知有关。
  • 5. 服务稳定性不是主要原因 近 7 天只有 1 条错误日志，高失败率用户为 0。已完成首次调用的用户总体留存和使用都较好。

运营行动建议

优先提高首次成功调用率，再处理支付召回和高价值用户留存。

  • P0补齐新手激活闭环 给新用户一个“创建 Key -> 复制 Base URL -> 选择推荐模型 -> 一键复制 curl / 客户端配置 -> 看到首次成功请求”的路径。首屏不要只展示概念，直接让用户完成一次可验证调用。
  • P0配置小额新手试用额度 建议设置 QuotaForNewUser 或 QuotaForInvitee，让直注册用户能完成至少一次真实调用。额度不必大，目标是把首次成功调用率从 31.1% 拉升到 45% 以上。
  • P1按用户状态做召回 对“无 Key”用户推创建 Key 指引；对“有 Key 无消费”用户推 Base URL、模型名、客户端导入教程；对“支付 expired”用户推继续支付/重新下单入口。
  • P1优化支付完成率 针对 54 笔 expired/failed 订单检查支付页文案、套餐权益、支付倒计时、回跳确认和 webhook 到账提示。近 7 天成功率已经到 64.3%，目标可以先定到 70%。
  • P2维护头部高价值用户 Top 10 用户贡献 63.02% 的近 7 天消耗，需要建立余额提醒、订阅额度提醒、专属模型稳定性监控，避免头部用户因额度或模型变动中断。

补充排查：客户主题 Turnstile

Turnstile 开启后的登录不稳定主要发生在客户主题登录链路，不属于模型调用服务稳定性问题。

  • 1. 运营影响 Turnstile 如果在注册、登录、邮箱验证码、OAuth、Passkey 等入口不稳定，会直接放大新用户注册后的首登和首次使用损耗，尤其影响报告中 0-7 天新增用户的激活转化。
  • 2. 主要原因 客户主题原先把 Turnstile token 放在 URL query 中传递，生产访问日志会记录较长的 token；注册成功后还会立即复用同一个 token 做自动登录。Turnstile token 具备短时效和一次性校验特征，复用会造成“有时成功、有时失败”的体验。
  • 3. 已处理方向 客户主题已改为通过 X-Turnstile-Token 请求头传递 token，后端中间件兼容该 header 并保留 query 回退；后端校验请求增加 8 秒超时，避免 Cloudflare 校验异常时拖住登录请求。
  • 4. 后续观察 重新开启 Turnstile 后建议重点观察登录失败率、注册成功后自动登录成功率、验证码发送成功率，以及 0-1 天新增用户的创建 Key 转化率是否回升。
